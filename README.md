# ClassicMCP

Автоматизация лабораторных работ по ТАУ (Теория автоматического управления) в программе **CLASSiC 3.2** через MCP-сервер для Claude Code.

---

## Быстрый старт

### 1. Установка зависимостей

```bash
cd classic_automation
pip install -r requirements.txt
```

### 2. Регистрация MCP-сервера в Claude Code

```bash
claude mcp add classic-automation python classic_automation/server.py
```

Или добавь вручную в `.claude/settings.json` проекта:

```json
{
  "mcpServers": {
    "classic-automation": {
      "command": "python",
      "args": ["classic_automation/server.py"]
    }
  }
}
```

### 3. Проверка

Запусти в Claude Code:

> "Проверь зависимости classic-automation"

Claude вызовет инструмент `check_dependencies` и покажет статус каждого компонента.

### 4. Подготовка шаблона

В папке `classic/` должен лежать файл `template.mdl` — модель варианта 17, созданная вручную в CLASSiC 3.2 с параметрами:

| Блок | Передаточная функция | Параметры |
|------|----------------------|-----------|
| W2   | K1                   | K1 = 100  |
| W3   | K3 / (T3·s + 1)     | K3 = 2.5, T3 = 0.5 |
| W4   | K4 / (T4·s + 1)     | K4 = 0.2, T4 = 0.05 |
| W5   | K5 / s              | K5 = 0.05 |

## Использование

Типичный диалог с Claude Code:

> "Обработай задание tasks/LP_v17.doc"

Claude сам выберет нужные инструменты в правильном порядке. Можно также вызывать инструменты по отдельности:

> "Рассчитай передаточные функции для варианта 17"  
> "Сгенерируй MDL для варианта 3"  
> "Запусти CLASSiC и сними скриншоты для var17.mdl"

## Структура проекта

```
ClassicMCP/
├── classic/                  — CLASSiC 3.2 (Win16 через otvdm)
│   ├── CLASSiC32.exe
│   ├── CLASSiC.ini
│   ├── template.mdl          — шаблон модели (создаётся вручную)
│   └── vdm/                  — otvdm (эмулятор Win16)
├── classic_automation/       — MCP-сервер и все модули
│   ├── server.py             — FastMCP-сервер (точка входа)
│   ├── config.py             — пути (вычисляются автоматически)
│   ├── parser.py             — парсинг .doc, таблица вариантов
│   ├── calculator.py         — расчёты через SymPy
│   ├── mdl_writer.py         — генерация .mdl файла
│   ├── mdl_patcher.py        — бинарный патч template.mdl
│   ├── classic_gui.py        — автоматизация CLASSiC через pyautogui
│   ├── report.py             — заполнение .docx отчёта
│   └── main.py               — CLI-запуск без MCP
├── tasks/                    — .doc файлы заданий
├── models/                   — сгенерированные .mdl файлы
└── reports/                  — готовые отчёты и скриншоты
```

## Доступные инструменты MCP

| Инструмент | Что делает |
|------------|------------|
| `check_dependencies` | Проверяет Python-пакеты, CLASSiC.exe, рабочие папки |
| `list_task_files` | Перечисляет .doc/.docx файлы в папке заданий |
| `parse_task_file` | Читает .doc, извлекает номер варианта и параметры |
| `get_variant_params` | Возвращает параметры варианта из таблицы (без файла) |
| `calculate_transfer_functions` | Считает WP(s), Ф(s), Фe(s), Гурвиц, K1кр, eуст |
| `write_mdl_file` | Генерирует .mdl модель для CLASSiC |
| `run_classic_gui` | Запускает CLASSiC, снимает 7 скриншотов |
| `fill_report_docx` | Вставляет расчёты и скриншоты в .docx шаблон |
| `run_full_pipeline` | Полный пайплайн одной командой (парсинг → отчёт) |

### Порядок вызовов

```
get_variant_params / parse_task_file
        ↓
calculate_transfer_functions
        ↓
write_mdl_file
        ↓
run_classic_gui
        ↓
fill_report_docx
```

---

## Что уже работает

- **Расчёты** (`calculate_transfer_functions`) — полностью. SymPy считает все передаточные функции, критерий Гурвица, критическое усиление K1кр, установившиеся ошибки.
- **Генерация MDL** (`write_mdl_file`) — полностью. Бинарный патч `template.mdl` работает для всех 20 вариантов.
- **Запуск CLASSiC и скриншоты** (`run_classic_gui`) — 7 из 8 скриншотов: схема, текстовая форма, характеристики (общий вид + 4 квадранта).
- **Парсинг** (`parse_task_file`) — работает для `.docx` и файлов с `v17` / `вариант 17` в имени или тексте. Для `.doc` требует LibreOffice.

## Что пока не работает / не проверено

- **Скриншот при K1кр** (`critical` в `run_classic_gui`) — автоматический двойной клик по блоку K1 для изменения параметра не откалиброван. Координата блока задана приблизительно (`0.28, 0.50` от размера окна).
- **Заполнение отчёта** (`fill_report_docx`) — модуль написан, но не тестировался end-to-end. Требует `.docx`-шаблон с плейсхолдерами вида `[TASK4_WP]`, `[IMG:schema]` и т.д.
- **Парсинг `.doc`** — без LibreOffice работает только если номер варианта есть в имени файла.

## Известные ограничения

- **Только этот ПК / сервер с CLASSiC**: `run_classic_gui` требует установленного CLASSiC 3.2 и `template.mdl`. На машине без GUI (headless-сервер) не работает.
- **Мониторы**: при запуске CLASSiC может открыться на вторичном мониторе — скрипт автоматически перемещает окно на основной (координаты `200, 150`). Если основной монитор другой, поправь в `classic_gui.py` → `_move_to_primary(x=..., y=...)`.
- **Количество вариантов**: в `parser.py → VARIANT_TABLE` сейчас 20 вариантов с примерными параметрами. Уточни свою таблицу вариантов.
- **Экранное разрешение**: проверено на 3440×1440 (125% DPI). На других разрешениях меню CLASSiC может потребовать корректировки `_MRU_DOWNS` в `classic_gui.py`.
